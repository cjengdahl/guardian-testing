# AI Code Assistant Security Guidelines

- Review and remove committed secrets; require rotation and repo scrub for any detected credentials before merging.
- Enforce secret-management patterns in generated code: no hardcoded credentials; use environment variables or secret managers with clear placeholders.
- Require network calls generated by the assistant to include explicit timeouts and robust error handling.
- Add CI/static-scan gates for secrets and insecure patterns (detect-secrets/trufflehog, bandit, custom rules) that block merges until fixed.
- Provide secure code templates and tests the assistant must use when generating examples or scaffolding.

## Purpose / Scope
These instructions guide the AI Code Assistant tool on secure coding and repository hygiene to prevent generation of code that contains hardcoded credentials or network calls without timeouts (addressing B105 and B113). They apply to all generated code, examples, tests, and configuration files.

## High-level Policies for Generated Code
- Never emit real secrets, API keys, passwords, or private keys. Always use clearly-labeled placeholders (e.g., <SECRET_NAME>, REDACTED_SECRET) or environment variable names (e.g., os.environ["DB_PASSWORD"]).
- Never include values that resemble real credentials (e.g., AWS-like strings, realistic password patterns). If an example requires a secret format, use obviously fake placeholders (EXAMPLE_KEY_XXXX).
- All outbound network calls must include explicit timeouts and reasonable retry/backoff behavior. Do not rely on implicit library defaults.
- Generated code that needs configuration should use environment variables, configuration files loaded via a config library, or documented secret-manager integrations. Provide examples only for secure retrieval (env vars or secret manager SDKs), not inline values.
- When a user asks to regenerate or reproduce an exact previously leaked secret, refuse and instead provide remediation guidance.

## Secret Management (B105)
- Preferred order for secrets in generated code:
  1. Secret Manager (AWS Secrets Manager, Azure Key Vault, GCP Secret Manager) — show SDK usage with placeholder secret name.
  2. Environment variables (os.environ, dotenv) — demonstrate loading and fall back checks.
  3. Encrypted config files (if used, show how to decrypt via KMS/secret manager).
- Example: retrieving from environment (Python)
```python
import os
from typing import Optional

def get_db_password() -> Optional[str]:
    # Use environment variables; do not hardcode credentials.
    return os.environ.get("DB_PASSWORD")  # placeholder: set DB_PASSWORD in secret store/CI
```
- Example: retrieving from AWS Secrets Manager (Python)
```python
import boto3
import json
from botocore.exceptions import ClientError

def get_secret(secret_name: str) -> dict:
    # DO NOT embed secret values here. secret_name should be a placeholder like "my-app/db-password"
    client = boto3.client("secretsmanager", region_name="us-east-1")
    try:
        resp = client.get_secret_value(SecretId=secret_name)
        secret_string = resp.get("SecretString")
        return json.loads(secret_string) if secret_string else {}
    except ClientError:
        raise
```
- Examples and tests must use placeholders like "<DB_PASSWORD_PLACEHOLDER>" or environment variable names, never literal secrets.
- If a scan finds hardcoded secrets in the repo:
  - Immediately rotate the exposed secret(s).
  - Remove the secret from git history (use git filter-repo or BFG) and push a scrubbed history branch.
  - Document the rotation and remediation steps in the PR description.

## Network Calls & Timeouts (B113)
- All generated code that performs HTTP requests must include explicit timeout values and error handling. Prefer separate connect/read timeouts on supported libraries.
- Example: Python requests with timeout and retries
```python
import requests
from requests.adapters import HTTPAdapter
from requests.packages.urllib3.util.retry import Retry

session = requests.Session()
retries = Retry(total=3, backoff_factor=0.5, status_forcelist=[502, 503, 504])
session.mount("https://", HTTPAdapter(max_retries=retries))

def fetch(url: str, timeout: float = 5.0):
    # Use explicit timeout (float = total seconds or (connect, read))
    try:
        resp = session.get(url, timeout=(3.0, 10.0))
        resp.raise_for_status()
        return resp.text
    except requests.RequestException as exc:
        # Handle/propagate error clearly
        raise
```
- If the target platform or library does not support separate connect/read timeouts, always use a finite overall timeout and document recommended values.
- For async libraries, show examples with timeouts appropriate for that framework (e.g., aiohttp with ClientTimeout).

## AI Code Assistant Generation Rules (must be enforced by the tool)
- Never produce code that contains:
  - Literal passwords, keys, tokens, or anything matching known secret patterns.
  - Realistic-looking examples of third-party API keys or cloud credentials (e.g., patterns similar to AWS access keys).
- When requested to provide examples containing credentials, respond with:
  - A prior warning (explain security reasons), and
  - Use placeholders only (ENV_VAR names or clearly fake strings).
- If the user explicitly requests "example credentials for testing", provide obvious dummy values marked REDACTED or EXAMPLE and ensure the assistant suggests using a test-only secret store.
- Always include brief comments in generated code pointing to the secret retrieval method (env var or secret manager).
- If generating a codebase scaffolding or templates, include a README note instructing contributors to configure secrets securely and not to commit them.

## CI / Pre-merge Scans and Enforcement
- Generated PRs must pass the following checks (configure CI to fail when these detect issues):
  - Secret detection: detect-secrets, truffleHog, or equivalent. Configure to scan committed files and block merges on findings.
  - Static analysis: bandit (or language equivalent) with checks for hardcoded credentials and unsafe network usage; fail on B105/B113-equivalent rules.
  - Lint/format: pre-commit with policies that include detect-secrets and bandit hooks.
- Example GitHub Actions job (minimal illustrative snippet):
```yaml
name: security-scan
on: [pull_request]
jobs:
  scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Install tooling
        run: |
          pip install detect-secrets bandit
      - name: Run detect-secrets
        run: detect-secrets-hook --baseline .secrets.baseline || true
      - name: Run bandit
        run: bandit -r . -lll -ii || exit $?
```
- Ensure the CI job is authoritative: deny merges until issues are resolved and show remediation guidance in CI output.

## Tests, Example Data, and Documentation
- Unit tests must never assert on or include real secrets. Use environment-variable injection or test-only fixtures that obtain secrets from a CI-managed secret store.
- Example configuration files in repo must use placeholders:
```env
# .env.example
DB_USER=example_user
DB_PASSWORD=<DB_PASSWORD_PLACEHOLDER>
```
- README should include a "Secrets and Configuration" section that instructs developers to create local dev secrets via a secure method (e.g., local vault, dev env variables) and to load them from .env (excluded from git).

## Remediation Guidance (If secrets are detected)
- Remove secrets from the codebase and git history; rotate credentials immediately with the secret provider.
- Create an incident note in the PR describing which credentials were rotated and which systems were affected.
- Re-run scans after history rewrite to confirm no remaining exposures.

## Implementation Checklist for Assistant Outputs
When producing or modifying code, the assistant must:
- Replace any literal secret it detects with a placeholder and add a comment pointing to the secret source (env var or secret manager).
- Insert explicit timeouts for all HTTP calls and a short example of retry/backoff handling.
- Add a short comment or README line explaining how to configure the secret in CI/secret manager.
- Suggest CI configuration to run detect-secrets and bandit and to fail PRs on findings.
- If the assistant's prior output (in the same session) included a secret, proactively notify the user and provide remediation steps instead of re-issuing the secret.

## Minimal Safe Templates (for quick reuse)
- Python env var template:
```python
import os

SECRET = os.environ.get("MY_SERVICE_SECRET")
if not SECRET:
    raise RuntimeError("MY_SERVICE_SECRET not set; see README for secure setup")
```
- Python requests template:
```python
import requests

def http_get(url: str, timeout=(3.0, 10.0)):
    try:
        resp = requests.get(url, timeout=timeout)
        resp.raise_for_status()
        return resp.json()
    except requests.RequestException:
        raise
```

## Notes for Maintainers
- Add detect-secrets baseline and pre-commit configuration to the repository.
- Configure CI to run the scans and fail on findings. Keep the secret baseline up to date and review it before adding any exceptions.
- Educate contributors about never committing .env with secrets; provide .env.example and a CONTRIBUTING.md snippet.

By following these rules the AI Code Assistant will avoid generating hardcoded secrets (B105) and network calls without timeouts (B113), and the repository will be protected via CI and scan enforcement.
